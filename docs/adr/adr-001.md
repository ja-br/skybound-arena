# ADR 0001 — Use ECS-native blue/green deployments instead of AWS CodeDeploy

- **Status:** Accepted
- **Date:** 2026-07-02
- **Deciders:** project owner
- **Supersedes:** the original CodeDeploy-owned deployment design in the compute module and CI/CD plan
- **Implementation status:** decision recorded; Terraform migration (compute module + removal of the pipeline/CodeDeploy module) tracked separately.

## Context

The project's deployment strategy is not incidental — blue/green with health-gated traffic shifting and automatic rollback is a core deliverable and the demo. The original design implemented it with **AWS CodeDeploy**: the ECS service ran under the `CODE_DEPLOY` deployment controller, Terraform stood up a bootstrap task revision on the blue target group and then relinquished `task_definition`, `load_balancer`, and `desired_count`, and CodeDeploy owned every subsequent traffic shift between blue and green target groups.

**ECS does blue/green natively.** On **2025-07-17** AWS announced built-in blue/green deployments for Amazon ECS ("Amazon ECS enables built-in blue/green deployments"). The ECS service itself can now provision a new (green) task set alongside the running (blue) one, validate it against a test listener with lifecycle hooks, shift production traffic, bake, and automatically roll back on a CloudWatch alarm or the deployment circuit breaker without CodeDeploy.


- `deployment_configuration { strategy = "BLUE_GREEN", bake_time_in_minutes }` plus `canary_configuration`, `linear_configuration`, and `lifecycle_hook` sub-blocks.
- `load_balancer { advanced_configuration { alternate_target_group_arn, production_listener_rule, test_listener_rule, role_arn } }`.

## Decision

**Adopt ECS-native blue/green deployment and remove AWS CodeDeploy from the project entirely.**

- The ECS service uses the default `ECS` deployment controller with a `deployment_configuration` of `strategy = "BLUE_GREEN"` and a configurable `bake_time_in_minutes`.
- Traffic shifting is driven through ALB listener rules: a production listener rule (blue⇄green) and an optional test listener rule for pre-shift validation. The service's `load_balancer.advanced_configuration` references the alternate (green) target group, the listener rules, and an ECS infrastructure IAM role.
- A new ECS infrastructure IAM role (AWS-managed policy `AmazonECSInfrastructureRolePolicyForLoadBalancers`, IAM being free) grants ECS permission to modify the target groups and listener rules during a shift.
- Terraform owns the task definition again.** Because a deployment is triggered by updating the service's task definition, the CodeDeploy-era `ignore_changes` on `task_definition`, `load_balancer`, and `desired_count` are removed. Updating the image is the deploy.
- Rollback is configured via the deployment circuit breaker and/or CloudWatch `deployment_alarms`, replacing the CodeDeploy auto-rollback configuration.
- The `modules/pipeline` module (CodeDeploy application + deployment group + service role) and `app/appspec.yaml` are deleted — both are CodeDeploy-specific and now dead.
- The future CI/CD app pipeline's deploy stage updates the ECS service with the new task definition (e.g. an ECS deploy action / `aws ecs update-service`) rather than handing an `appspec.yaml` to a `CodeDeployToECS` action.

The `/version` proof chain, the two target groups, `/healthz` health gating, and the "record a broken build → watch it auto-roll-back" demo are all unchanged, now serviced by ECS instead of CodeDeploy.

## Consequences

**Positive**

- **Simpler model.** One service owns the whole deploy. No cross-service handoff, no separate CodeDeploy application/deployment group, no `appspec.yaml`, no bootstrap-vs-CodeDeploy ownership split to explain.
- **Terraform regains ownership of the task definition**, eliminating the `ignore_changes` drift that the CodeDeploy design required.
- **Current-generation approach.** Reflects the direction AWS is steering ECS users, rather than the older CodeDeploy pattern.

**Negative / risks**

- **Listener-rule wiring.** The ALB is now wired via listener *rules* rather than a listener default action, an increase in ALB configuration.
- **Custom validation is Lambda-based.** Lifecycle-hook validation runs as Lambda functions, versus CodeDeploy's hook model.
